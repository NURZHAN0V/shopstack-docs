# Версии документации

Портал хранит **снимки документации по версиям API**, а не только «последний main». Каждая версия — отдельная папка в URL: `/v1.0/ru/...`, `/v1.0/en/...`.

## Текущие версии

| Версия docs | Контракт API | Статус | Папка |
|-------------|--------------|--------|-------|
| **v1.0** | 1.0 | актуальная | `docs/v1.0/` |

Переключатель в шапке сайта меняет сегмент версии в URL и сохраняет текущую страницу и локаль (если она есть в целевой версии).

## Когда заводить новую версию

Создавайте **v2.0** (или следующий minor), когда:

- в REST API появляются **breaking changes** (удаление полей, смена семантики, несовместимые пути);
- гайды панели или витрины описывают поведение, которое **больше не совпадает** с релизом для старых клиентов.

Мелкие правки и новые эндпоинты без breaking changes остаются в **актуальной** версии docs (сейчас `v1.0`).

## Как выпустить новую версию (чеклист)

1. Скопировать снимок: `pnpm version:snapshot v2.0` (из `v1.0`).
2. Добавить запись в `docs/.vitepress/config/versions.ts` (`key`, `apiContract`, `status`).
3. Пометить предыдущую версию как `legacy`, новую — `current`.
4. Обновить матрицу совместимости и changelog портала.
5. Внести правки только в `docs/v2.0/`; `docs/v1.0/` **замораживается** (исправления — опечатки, безопасность).
6. Создать git-тег `docs-v2.0.0` и GitHub Release (опционально).
7. Проверить `pnpm build` и переключатель версий на staging.

## Структура репозитория

```
docs/
  v1.0/ru/   v1.0/en/   # снимок для API 1.0
  v2.0/...               # следующий major (когда появится)
  .vitepress/config/
    versions.ts          # реестр версий (единый источник)
    paths.ts             # префикс актуальной версии для nav/sidebar
```
